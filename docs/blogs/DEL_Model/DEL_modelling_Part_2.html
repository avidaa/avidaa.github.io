Modelling DNA Encoded Libraries (DELs) - Part 2

What Is the Aim of This Blog?

This blog builds on the statistical foundations established in Part 1 by specifying and fitting a probabilistic model for DNA-encoded library (DEL) sequencing counts. The goal is to explain noisy observed counts in terms of latent enrichment, background matrix effects, input-library composition, and replicate-level technical variation, so that binding-related signal can be separated from technical and compositional artifacts. A compound-aware model that incorporates molecular structure is deferred to Part 3.

The causal sketch from Part 1

The causal sketch introduced in Part 1 is the starting point for the model building in Part 2. Its role is to tell us which sources of variation should be represented explicitly, rather than being left for the model to absorb in an ad hoc way.

In Part 1, the sketch identified four main causes of observed DEL counts:

  • Library composition,
  • Matrix background,
  • Target-specific binding,
  • Replicate-level technical noise.

In Part 2, each of these becomes a concrete part of the probabilistic model:

  • Input abundance explains baseline counts,
  • Matrix effects explain non-specific background,
  • Target-specific terms capture enrichment beyond background, and
  • Replicate terms account for technical variation across experiments.

This is the key modeling idea for the rest of the post: observed counts are treated as noisy consequences of several overlapping latent processes, not as direct measurements of binding. We follow a Bayesian workflow inspired by McElreath’s Statistical Rethinking to build and evaluate candidate models that respect the causal structure of the sketch while differing in their statistical assumptions.

A Bayesian workflow for modelling DEL counts

A DAG does not uniquely determine a statistical model. It tells us which causal relationships we are assuming, which paths should be blocked or left open, and which variables should or should not be conditioned on. But it does not choose the likelihood, link function, priors, hierarchy, or dispersion model. Those choices define the actual probabilistic model.

For DEL counts, the workflow is:

  1. State the causal assumptions (DAG first).
    Decide which processes generate the counts: input abundance, matrix background, target-specific enrichment, and replicate-level technical variation. Use the DAG to identify confounders, mediators, and colliders before writing a model.

  2. Write the generative model.
    Translate the DAG into a likelihood, link function, priors, and deterministic relationships between latent quantities and observed counts. Several models can be compatible with the same DAG, for example Negative Binomial versus Poisson-lognormal counts, different pooling structures, or different dispersion assumptions.

  3. Simulate from the assumed data-generating process.
    Before fitting the model to real data, simulate synthetic DEL counts from the model. This is where we check whether the assumed process can produce data that look like plausible DEL data. The simulation should include the same components we claim are present in the experiment: library abundance, matrix background, target enrichment, count noise, and replicate effects.

    This step answers a different question from model fitting: if this model were true, what kind of data would it generate? If the simulated counts are obviously unrealistic, the model is not ready to be fit.

  4. Fit the model to observed data.
    Estimate the posterior distribution for the latent quantities and parameters. For large DEL datasets this may require scalable inference, while smaller examples can use MCMC.

  5. Check the fitted model.
    Use posterior summaries and posterior predictive checks to ask whether the fitted model reproduces the observed data. Important checks include count dispersion, tail behaviour, replicate-to-replicate variation, and matrix-target relationships.

  6. Compare and revise.
    Compare candidate models using posterior predictive checks and, where useful, WAIC, LOO, or stacking. Good prediction is not enough on its own: the model also has to preserve the causal interpretation implied by the DAG.

  7. Interpret the target-binding signal.
    Once a model is both causally coherent and predictively adequate, use the posterior to interpret target-binding propensity, uncertainty, and causal contrasts justified by the DAG.

State the causal assumptions (DAG first)

The first step in the Bayesian workflow is to make the causal assumptions explicit as a directed acyclic graph (DAG). In our DAG, an observation is represented as a square node, a latent variable (unobserved) as a circle, and causal relationships are represented as directed edges between nodes. We build the DAG incrementally, starting from what is measured and then introducing the latent quantities needed to explain it.

For a single compound, the experiment yields three observed counts:

  • Matrix counts, \(C_m\),
  • Target counts, \(C_t\),
  • Library counts (pre-selection), \(C_l\).

These are shown as the only nodes in the initial DAG:

The library counts \(C_l\) are measured before any selection. The matrix counts \(C_m\) are measured after the library is exposed to the experimental matrix, so they are influenced by input abundance. The target counts \(C_t\) are measured after the library is exposed to both the matrix and the target, so they are influenced by input abundance and matrix background.

NoteWhat does valid inference mean in the context of a DAG?

Valid inference means that the statistical model allows us to recover the latent target-binding propensity \(s\) from the observed counts in a way that is consistent with the causal assumptions encoded in the DAG. In practice, this means accounting properly for other causes of the target counts, such as:

  • Input abundance \(a\),
  • Matrix background \(b\),
  • Replicate-level variation \(\gamma_m\) and \(\gamma_t\),

and avoiding conditioning choices that introduce bias, such as conditioning on colliders or leaving backdoor paths open.

Invalid inference occurs when the model attributes variation to \(s_i\) that is actually due to other pathways, or, when \(s_i\) is not identifiable from the observed data under the assumed model.

A DAG helps you reason about bias from conditioning, but valid inference also depends on model identification and likelihood specification. So even with the right DAG, inference can still be invalid if the statistical model is misspecified or the latent parameters are not identifiable.

Which variables should and should not be conditioned on?

Our goal is to infer the target-binding propensity \(s\). To do this, we need to understand how information about \(s\) flows through the DAG. This is where the concepts of path, open and closed paths, four elemental confounds (as described by McElreath - second edition of statistical rethinking, Figure 6.6) come in.

What is a path in a DAG?
What does open or closed path mean in a DAG?

A path is open when information can pass along it, meaning the two variables may become statistically associated.

A path is closed when something blocks that route, so the variables are independent along that path.

Whether a path is open or closed depends on two things:

  1. The arrow pattern along the path, and
  2. Which variables we condition on.

Opening and closing paths matter because conditioning can change the associations implied by a DAG. Closing an unwanted path, such as a confounding path, can reduce bias. But conditioning can also do harm: it may block part of the causal effect we want to estimate, or open a previously closed path and create a spurious association.

McElreath shows that these changes in path status are governed by a small set of recurring path structures. In practice, DAG reasoning means identifying which structure appears along a path and then asking whether conditioning on a node blocks bias or introduces it.

This is why simply removing correlated predictors is not a causal solution. As McElreath puts it, “Pairwise correlations are not the problem.” What matters is why variables are correlated and what conditioning on them does. A high correlation may arise because two variables share a common cause, because one mediates the effect of the other, or because both are downstream of some selection process. These situations require different modelling choices.

What are the recurring path structures and how they can open or close a path?

There are mainly four elemental path structures that determine how conditioning affects the flow of information in a DAG.

The first is the fork, or confounding, structure:

In a fork structure, the middle node is a common cause of the two endpoints:

\[X \leftarrow Z \rightarrow Y\]

Because both \(X\) and \(Y\) are caused by \(Z\), they may be statistically associated even if there is no direct causal effect between them. Conditioning on the common cause \(Z\) blocks this path and removes the association induced by \(Z\). This can be written as:

\[X \perp\!\!\!\perp Y \mid Z\]

which reads as “\(X\) is independent of \(Y\) once we know \(Z\)”.

Conditioning on \(Z\) blocks the backdoor path through the confounder, \(Z\), and helps recover an unbiased estimate of the causal effect.

NoteWhat is a backdoor path?

A backdoor path is a non-causal path between \(X\) and \(Y\) that sneaks in through a common cause and can make \(X\) look associated with \(Y\) even when the association is partly or entirely confounded.

A front-door causal path is: \[X \rightarrow Y\]

This is the direct causal effect we want to estimate: the effect of \(X\) on \(Y\).

Backdoor path start by going backwards out of \(X\) into one of its causes (in this case, \(Z\)), then forward to \(Y\):

\[X \leftarrow Z \rightarrow Y\]

Recall that a path is a sequence of edges connecting two nodes, regardless of the direction of the arrows. So the path from \(X\) to \(Y\) can go backwards from \(X\) to \(Z\), and then forward from \(Z\) to \(Y\). This is a backdoor path because it goes “backwards” out of \(X\) before going forward to \(Y\).

A backdoor path carries confounding. To estimate the causal effect of \(X\) on \(Y\), we usually want to block backdoor paths by conditioning on appropriate common causes such as \(Z\).

TipIntuitive example of a fork structure and backdoor path

Suppose we want to estimate whether carrying an umbrella has an effect on being wet. However, there is a confounder: weather conditions. Rainy weather affects both whether someone carries an umbrella and whether they gets wet.

On rainy days, people are more likely to carry an umbrella and more likely to get wet. This can create a statistical association between umbrellas and getting wet, even if carrying an umbrella does not directly cause someone to get wet.

\[X = \text{Carrying an umbrella}\]

\[Y = \text{Getting wet}\]

\[Z = \text{Weather conditions (rainy or not)}\]

Weather is a common cause of both carrying an umbrella and getting wet. This creates the backdoor path:

\[X \leftarrow Z \rightarrow Y\]

or \[\text{Carrying an umbrella} \leftarrow \text{Weather conditions} \rightarrow \text{Getting wet}\]

So the association between umbrella use and getting wet is partly coming through the backdoor path via weather.

The path is called a backdoor path because instead of going forward from umbrella to getting wet: \[\text{Carrying an umbrella} \rightarrow \text{Getting wet}\]

It goes backwards from umbrella to one of its causes (weather):

\[\text{Carrying an umbrella} \leftarrow \text{Weather conditions}\]

and then forward from weather to getting wet: \[\text{Weather conditions} \rightarrow \text{Getting wet}\]

If we condition on weather, we compare people under the same weather conditions. This makes the causal question clearer. Weather creates a backdoor path because it affects both whether someone carries an umbrella and whether they get wet:

\[\text{Carrying an umbrella} \leftarrow \text{Weather} \rightarrow \text{Getting wet}\]

Conditioning on weather blocks this backdoor path. It separates the association caused by shared weather conditions from the possible direct effect of carrying an umbrella on getting wet. Once we compare people under the same weather conditions, we would expect umbrella use to be associated with a lower chance of getting wet, if umbrellas are effective.

This code simulates a dataset of 1000 observations with three variables: Weather, Umbrella, and GettingWet.

Weather

Weather is simulated as a binary variable where 1 indicates rainy weather and 0 indicates not rainy. The probability of it being rainy is set to 0.3, meaning that 30% of the time it will be rainy.

Umbrella

The probability of carrying an umbrella is influenced by the weather and is defined as: \[ 0.8 * weather + 0.1 \]

This means that:

  • If it’s rainy (weather = 1), the probability of carrying an umbrella is 90%: \[ 0.8 * 1 + 0.1 = 0.9 \]

  • If it’s not rainy (weather = 0), the probability of carrying an umbrella is 10%: \[ 0.8 * 0 + 0.1 = 0.1 \]

Getting Wet

The probability of getting wet is defined as: \[ 0.7 * weather * (1 - umbrella) + 0.1 * weather * umbrella + 0.05 * (1 - weather) \]

This means that:

  • If it’s rainy and the person is not carrying an umbrella (weather = 1, umbrella = 0), the probability of getting wet is 70%: \[ 0.7 * 1 * (1 - 0) + 0.1 * 1 * 0 + 0.05 * (1 - 1) = 0.7 \]

  • If it’s rainy and the person is carrying an umbrella (weather = 1, umbrella = 1), the probability of getting wet is 10%: \[ 0.7 * 1 * (1 - 1) + 0.1 * 1 * 1 + 0.05 * (1 - 1) = 0.1 \]

  • If it’s not rainy (weather = 0), the probability of getting wet is always low at 5%, regardless of umbrella use. \[ 0.7 * 0 * (1 - umbrella) + 0.1 * 0 * umbrella + 0.05 * (1 - 0) = 0.05 \]

Analyze the data to see the associations

We are interested in the association between carrying an umbrella and getting wet, and how this association changes when we condition on weather.

To quantify how the umbrella–wet association changes when we account for weather, we fit two logistic regression models: one unadjusted (umbrella only) and one adjusted (umbrella and weather).

Both models treat each person’s outcome as a Bernoulli draw with an individual probability \(p_i\):

\[\text{GettingWet}_i \sim \text{Bernoulli}(p_i)\]

The two models differ in how \(p_i\) is parameterised. The unadjusted model links \(p_i\) only to umbrella use:

\[\text{logit}(p_i) = \alpha + \beta_{\text{umbrella}} \cdot \text{Umbrella}_i\]

The adjusted model additionally conditions on weather, blocking the backdoor path \(\text{Umbrella} \leftarrow \text{Weather} \rightarrow \text{GettingWet}\):

\[\text{logit}(p_i) = \alpha + \beta_{\text{umbrella}} \cdot \text{Umbrella}_i + \beta_{\text{weather}} \cdot \text{Weather}_i\]

The priors are weakly informative Normal(0, 2) distributions on each coefficient, which keep the logit-scale effects within a plausible range (\(\pm 4\)) while placing no strong prior preference on direction:

\[ \alpha \sim \text{Normal}(0, 2), \quad \beta_{\text{umbrella}} \sim \text{Normal}(0, 2), \quad \beta_{\text{weather}} \sim \text{Normal}(0, 2) \]

Both models are fitted via NUTS (No-U-Turn Sampler) using Pyro. The key quantity of interest is \(\beta_{\text{umbrella}}\): comparing its posterior between the two models shows how much of the raw umbrella–wet association disappears once weather is held fixed.

Code 
import warnings
import torch
import pyro
import pyro.distributions as dist
from pyro.infer import MCMC, NUTS

warnings.filterwarnings("ignore", category=UserWarning)


pyro.set_rng_seed(42)


def wet_model(umbrella, wet, weather=None):
    alpha = pyro.sample('alpha', dist.Normal(0.0, 2.0))
    beta_umbrella = pyro.sample('beta_umbrella', dist.Normal(0.0, 2.0))

    logits = alpha + beta_umbrella * umbrella
    if weather is not None:
        beta_weather = pyro.sample('beta_weather', dist.Normal(0.0, 2.0))
        logits = logits + beta_weather * weather

    with pyro.plate('obs', len(umbrella)):
        pyro.sample('wet', dist.Bernoulli(logits=logits), obs=wet)


def fit_logistic_model(umbrella, wet, weather=None, num_samples=1000, warmup_steps=500):
    pyro.clear_param_store()
    umbrella = torch.as_tensor(umbrella, dtype=torch.float32)
    wet = torch.as_tensor(wet, dtype=torch.float32)
    weather = None if weather is None else torch.as_tensor(weather, dtype=torch.float32)

    nuts_kernel = NUTS(wet_model)
    mcmc = MCMC(
        nuts_kernel,
        num_samples=num_samples,
        warmup_steps=warmup_steps,
        disable_progbar=True
    )
    mcmc.run(umbrella, wet, weather)
    return mcmc.get_samples()

The posteriors tell a clear story about confounding. In the unadjusted model, \(\beta_{\text{umbrella}}\) is centred near zero — naively suggesting that carrying an umbrella makes no difference to getting wet. This is the confound at work: rainy weather drives both umbrella use and getting wet, so the two variables appear unrelated once that shared cause dominates the marginal association.

In the adjusted model, conditioning on weather blocks the backdoor path \(\text{Umbrella} \leftarrow \text{Weather} \rightarrow \text{GettingWet}\), and \(\beta_{\text{umbrella}}\) shifts decisively to the left. A negative logit-scale coefficient means that, among people experiencing the same weather conditions, carrying an umbrella is associated with a lower probability of getting wet, exactly what the data-generating process encoded. The weather effect is itself strongly positive, confirming that rainy weather is the main driver of getting wet, independent of umbrella use.

This is the core lesson of confounding: an unadjusted association can be essentially zero even when the causal effect is real, because a common cause masks it. Conditioning on that cause separates the causal signal from the noise introduced by the shared pathway.

TipWhat if we only observe Umbrella and Getting Wet, but not Weather?

Up to this point, we assumed that Weather, Umbrella, and Getting Wet were all observed. In that case, the confounding structure is visible and can be blocked by conditioning on Weather:

\[\text{Umbrella} \leftarrow \text{Weather} \rightarrow \text{Getting Wet}.\]

Now suppose Weather is unobserved. The causal structure has not changed; the same fork still exists, but the common cause is now latent:

\[\text{Umbrella} \leftarrow \text{Weather}_{\text{latent}} \rightarrow \text{Getting Wet}.\]

The confounder does not disappear just because we failed to measure it. It still keeps the backdoor path open.

As a result, a model that regresses Getting Wet on Umbrella alone mixes two sources of association:

  1. The causal effect of carrying an umbrella, and
  2. The non-causal association induced by hidden Weather.

Because Weather is unobserved, ordinary adjustment cannot block this path directly. We therefore need another strategy. The most transparent first step is sensitivity analysis, which asks:

How large would unmeasured-confounding bias need to be to materially change the conclusion?

Other strategies are possible, but they require more data or more assumptions. We could measure Weather directly, use proxy variables such as rainfall or season, randomise umbrella use, or build a latent variable model that represents Weather explicitly. Each option addresses the same problem: the open backdoor path created by an unobserved common cause.

The code below starts with a simple sensitivity analysis, then fits a latent variable model in which Weather is unobserved but still included in the generative structure.

Latent confounders are dangerous because they keep backdoor paths open. If the common cause is unmeasured, the model can attribute part of a spurious association to the causal effect we want to estimate.

NoteSensitivity analysis and latent variable model
Sensitivity analysis for hidden confounding

We start with a simple sensitivity analysis. Figure 8 makes the causal problem explicit: hidden Weather is a common cause of both Umbrella and Getting Wet, so even if we observe those two variables perfectly, the umbrella coefficient from a model that omits Weather can mix together a causal effect and confounding bias.

To formalise this, we write the naive umbrella association as the sum of a causal component and an omitted-confounding term:

\[\beta_{\text{naive}} = \beta_{\text{causal}} + \delta,\]

where:

  • \(\beta_{\text{naive}}\) is the umbrella coefficient from the unadjusted model,
  • \(\beta_{\text{causal}}\) is the (unknown) causal umbrella effect we care about,
  • \(\delta \ge 0\) is an assumed positive bias induced by hidden Weather.
TipWhy a positive bias?

Hidden Weather is a common cause of both Umbrella and Getting Wet as Figure 8 shows. This means that:

  • When Weather = 1, umbrella use becomes more likely, and
  • When Weather = 1, getting wet also becomes more likely.

This shared cause induces an extra positive association between umbrella use and getting wet in the unadjusted model. In this sensitivity analysis, we represent that upward bias with \(\delta\), so we assume \(\delta \ge 0\).

The point is not that umbrellas truly increase wetness. The point is that hidden Weather can pull the naive umbrella coefficient upward, making the effect look less negative, closer to zero, or even positive.

That is why we write:

\[\beta_{\text{naive}} = \beta_{\text{causal}} + \delta\]

and therefore:

\[\beta_{\text{causal}}(\delta) = \beta_{\text{naive}} - \delta.\]

Rearranging gives the bias-corrected umbrella effect under an assumed bias level \(\delta\):

\[\beta_{\text{causal}}(\delta) = \beta_{\text{naive}} - \delta.\]

In Bayesian form, we apply this transformation to each posterior draw from the unadjusted model:

\[\beta_{\text{causal}}(\delta_i) = \beta_{\text{naive}} - \delta_i, \quad i=1,\dots,N.\]

For each value of \(\delta\) on a grid, we compute summary statistics of the adjusted posterior draws \(\{\beta_{\text{causal}}(\delta)_i\}_{i=1}^N\) and plot the median together with an 80% interval. Figure 9 shows the logic of this one-sided setup: as the assumed hidden-weather bias increases, the corrected umbrella effect is shifted downward by exactly that amount.

NoteWhy sensitivity analysis still matters when the unadjusted effect is zero?

Even if the unadjusted umbrella effect were exactly zero, we should not automatically ignore sensitivity analysis.

In this example, \(\beta_{\text{naive}}\) is already close to zero. Under the one-sided assumption that hidden Weather biases the umbrella coefficient upward, any positive value of \(\delta\) moves the corrected effect below zero:

\[\beta_{\text{causal}}(\delta)=\beta_{\text{naive}}-\delta.\]

So the sensitivity analysis is not testing whether a strong positive umbrella effect survives adjustment. It is showing that the unadjusted association is fragile around zero: even modest hidden-weather bias would make the corrected effect negative, consistent with umbrellas reducing the chance of getting wet.

However, this does not mean that sensitivity analysis is unnecessary whenever an observed coefficient is near zero and a common cause is plausible.

The direction and size of the hidden bias may be uncertain, the posterior uncertainty may be wide, and the causal question may depend on how large the corrected effect could plausibly become. Sensitivity analysis is still useful because it makes those assumptions explicit.

Latent variable model for hidden confounding

The sensitivity analysis treated hidden Weather as an unknown source of bias. That is useful because it asks: how much hidden confounding would be needed to change our conclusion? But it does not try to model Weather directly.

A stronger alternative is to write down a latent variable model. In this model, Weather is not just an abstract bias term. It is an explicit but unobserved variable that can affect both whether someone carries an umbrella and whether they get wet.

The assumed causal structure is as shown in Figure 6: Weather is a common cause of both Umbrella and Getting Wet, and there is also a direct effect of Umbrella on Getting Wet. However, only Umbrella and GettingWet are observed. Weather is hidden. This is similar to the DAG in Figure 8.

Let, \[W_i \in \{0, 1\}\]

denote the latent weather state for observation \(i\), where \(W_i = 0\) might mean dry weather and \(W_i = 1\) might mean rainy weather. We observe:

\[ U_i = Umbrella_i \] \[ G_i = GettingWet_i \]

The model first assumes that each observation has some probability of being in rainy weather: \[W_i \sim \text{Bernoulli}(p_{\text{weather}}),\] Then, umbrella use depends on the hidden weather state: \[U_i|W_i \sim \text{Bernoulli}(\text{logit}^{-1 }(\alpha_u + \beta_{\text{weather} \to \text{umbrella}} \cdot W_i)),\] where, \(\alpha_u\) is the baseline log-odds of carrying an umbrella when \(W_i=0\), and \(\beta_{\text{weather} \to \text{umbrella}}\) captures how much the log-odds of carrying an umbrella increase when \(W_i=1\).

Getting wet then depends on both umbrella use and the hidden weather state: \[G_i|U_i, W_i \sim \text{Bernoulli}(\text{logit}^{-1 }(\alpha_g + \beta_{\text{umbrella} \to \text{wet}} \cdot U_i + \beta_{\text{weather} \to \text{wet}} \cdot W_i)),\] where \(\alpha_g\) is the baseline log-odds of getting wet when \(U_i=0\) and \(W_i=0\), \(\beta_{\text{umbrella} \to \text{wet}}\) captures the direct umbrella effect, after accounting for latent Weather, and \(\beta_{\text{weather} \to \text{wet}}\) captures the direct effect of Weather on getting wet, i.e., how much the log-odds of getting wet increase when \(W_i=1\).

The important computational detail is that we do not sample \(W_i\) directly. Weather is discrete, and NUTS works best with continuous parameters. So instead we marginalise over the two possible weather states. ​ For each observation, the likelihood is: \[P(U_i, G_i) = \Sigma_{w_i \in \{0, 1\}} P(W_i=w) P(U_i | W_i=w) P(G_i | U_i, W_i=w) = P(W_i=0) P(U_i | W_i=0) P(G_i | U_i, W_i=0) + P(W_i=1) P(U_i | W_i=1) P(G_i | U_i, W_i=1)\]

So the model asks:

For each observed pair \((U_i, G_i)\), how likely is this observation if the hidden weather state was dry, and how likely is it if the hidden weather state was rainy? It then averages those two possibilities according to the estimated probability of each weather state.

Because Weather is never observed, the model must infer it entirely from the joint pattern of Umbrella and GettingWet. That means the result depends strongly on the assumed model form, the priors, and the assumption that a binary latent Weather variable is a reasonable representation of the hidden common cause.

So the latent variable model gives us a more mechanistic version of the hidden-confounding story, but it is not assumption-free. The sensitivity analysis is weaker but more transparent; the latent variable model is more structural but more model-dependent.

def latent_weather_model(umbrella, wet):
    """
    Marginalized latent common-cause model:

        Weather -> Umbrella
        Weather -> GettingWet
        Umbrella -> GettingWet

    Only Umbrella and GettingWet are observed.
    Weather is binary and latent, so we integrate it out of the likelihood.
    """
    logit_p_weather = pyro.sample("logit_p_weather", dist.Normal(0.0, 1.5))
    p_weather = torch.sigmoid(logit_p_weather)

    alpha_u = pyro.sample("alpha_u", dist.Normal(0.0, 2.0))
    beta_weather_to_umbrella = pyro.sample("beta_weather_to_umbrella", dist.Normal(0.0, 2.0))

    alpha_wet = pyro.sample("alpha_wet", dist.Normal(0.0, 2.0))
    beta_umbrella_to_wet = pyro.sample("beta_umbrella_to_wet", dist.Normal(0.0, 2.0))
    beta_weather_to_wet = pyro.sample("beta_weather_to_wet", dist.Normal(0.0, 2.0))

    umbrella_log_prob_w0 = dist.Bernoulli(logits=alpha_u).log_prob(umbrella)
    umbrella_log_prob_w1 = dist.Bernoulli(
        logits=alpha_u + beta_weather_to_umbrella
    ).log_prob(umbrella)

    wet_log_prob_w0 = dist.Bernoulli(
        logits=alpha_wet + beta_umbrella_to_wet * umbrella
    ).log_prob(wet)
    wet_log_prob_w1 = dist.Bernoulli(
        logits=alpha_wet + beta_umbrella_to_wet * umbrella + beta_weather_to_wet
    ).log_prob(wet)

    # The two terms log_prob_w0 and log_prob_w1 are the log-probabilities of the observed data under the two possible hidden Weather states.
    log_prob_w0 = torch.log1p(-p_weather) + umbrella_log_prob_w0 + wet_log_prob_w0
    log_prob_w1 = torch.log(p_weather) + umbrella_log_prob_w1 + wet_log_prob_w1
    
    # The logsumexp operation adds these two possibilities together on the probability scale while keeping the computation numerically stable.
    pyro.factor(
        "marginalized_obs_log_prob",
        torch.logsumexp(torch.stack([log_prob_w0, log_prob_w1]), dim=0).sum(),
    )


def fit_latent_weather_model(data, num_samples=1000, warmup_steps=500):
    pyro.clear_param_store()

    umbrella = torch.as_tensor(data["Umbrella"].values, dtype=torch.float32)
    wet = torch.as_tensor(data["GettingWet"].values, dtype=torch.float32)

    nuts_kernel = NUTS(latent_weather_model)
    mcmc = MCMC(
        nuts_kernel,
        num_samples=num_samples,
        warmup_steps=warmup_steps,
        disable_progbar=True,
    )
    mcmc.run(umbrella, wet)
    return mcmc.get_samples()

The latent-variable model gives a more structural representation of the hidden-confounding story: Weather is treated as an explicit but unobserved common cause of both Umbrella and GettingWet. However, the posterior estimates show wide uncertainty intervals for all three paths, and each interval overlaps zero. This means the model does not clearly identify the hidden Weather mechanism or the direct effect of Umbrella on GettingWet from the observed data alone.

In particular, the Umbrella → GettingWet coefficient is centred close to zero, suggesting little evidence for a strong direct umbrella effect after accounting for latent Weather. But this should not be interpreted as proof of no causal effect. Rather, it shows that once we allow for an unobserved common cause, the available data are compatible with a range of explanations.

Latent-variable models can make hidden confounding explicit, but they require stronger assumptions. If the hidden variable is not observed, the parameters may be weakly identified, and the posterior can remain highly uncertain. In this example, the latent Weather model is more faithful to the causal DAG, but less transparent and more assumption-dependent than the sensitivity analysis.

NoteA Note on Parameter Identifiability

The weak identification we saw above is not unique to latent-variable models. It belongs to a broader class of difficulties that arise whenever the data and model structure are not informative enough to pin down a parameter’s value. This can happen because of a modelling mistake, but it also occurs in perfectly correct models.

A practical signal of weak identification is that the posterior looks a lot like the prior. If the data carry little information about a parameter, Bayesian updating has little to update, and the posterior will closely track whatever prior we started with. Comparing the two is therefore a useful diagnostic: a posterior that remains wide and prior-shaped is not a sign that something went wrong, it is the correct answer to the question we asked. But it is also an invitation to ask a sharper question, collect more data, or reconsider whether the parameter is estimable at all given the available evidence.

The second is the pipe or mediation structure:

In a pipe, \(Z\) lies on the causal path from \(X\) to \(Y\). It is a mediator, not a common cause.

\[X \rightarrow Z \rightarrow Y\]

Conditioning on \(Z\) blocks this mediation path (\(X \perp\!\!\!\perp Y \mid Z\)). That is helpful only if our goal is to estimate the direct effect of \(X\) on \(Y\) that does not run through \(Z\). If we want to estimate the total effect of \(X\) on \(Y\), then conditioning on \(Z\) is inappropriate because it removes part of the causal effect we are trying to measure.

To estimate the total effect of \(X\) on \(Y\), we do not condition on \(Z\), because the mediator carries part of the causal signal.

To estimate the direct effect of \(X\) on \(Y\) that bypasses \(Z\), we do condition on \(Z\). This is a different estimand and answers a different causal question.

TipIntuitive example of a pipe structure

Suppose we want to understand whether exercise affects blood pressure, and part of that effect works through weight loss:

\[X = \text{Exercise}\]

\[Z = \text{Weight loss}\]

\[Y = \text{Blood pressure}\]

The assumed causal path is:

\[\text{Exercise} \rightarrow \text{Weight loss} \rightarrow \text{Blood pressure}\]

Here, weight loss mediates part of the effect of exercise on blood pressure. Exercise changes body weight, and that weight change in turn affects blood pressure.

If we condition on weight loss, we block that mediated path. The model then asks a narrower question: does exercise affect blood pressure other than through weight loss? That is the direct effect of exercise that bypasses the mediator.

If instead we want the total effect of exercise on blood pressure, we should not condition on weight loss, because weight loss is one of the mechanisms through which exercise works. Conditioning on the mediator would remove that part of the causal signal and change the estimand.

NoteTotal effect vs direct effect vs mediated effect in the pipe structure

Total effect

The total effect is the overall effect of Exercise on Blood pressure, allowing the effect to travel through all available causal routes. In this pipe example, there is only one route:

\[\text{Exercise} \rightarrow \text{Weight loss} \rightarrow \text{Blood pressure}\]

So the total effect includes the full contribution of Weight loss as a mediator.

Interpretation: this asks, “if a person exercises rather than not exercising, how much does their blood pressure change overall?” To answer that question, we do not condition on Weight loss, because Weight loss is part of how Exercise works.

Direct effect

The direct effect is the effect of Exercise on Blood pressure after blocking the route through Weight loss. That means we ask whether Exercise still changes Blood pressure once Weight loss is held fixed.

Interpretation: this asks, “if two people had the same Weight loss, but one exercised and the other did not, would their Blood pressure still differ?” In the Code for the pipe example section, the simulated data-generating process says no. So in that example, the true direct effect is 0.

Mediated path

The mediated path is the route through Weight loss. It is the part of the story in which Exercise first changes Weight loss, and Weight loss then changes Blood pressure.

Interpretation: this panel is not asking about the overall effect of Exercise. It is isolating the mediator-to-outcome part of the pipe, namely how Blood pressure changes as Weight loss changes. Later, when we fit the Bayesian models, this is the quantity represented by the WeightLoss -> BP | Exercise coefficient.

These three pictures answer three different questions. Total effect asks for the overall effect of Exercise on Blood pressure. Direct effect asks what remains after Weight loss is held fixed. Mediated path isolates the WeightLoss -> Blood pressure part of the mechanism.

NoteCode for the pipe example

Let’s simulate some data to illustrate the pipe example. We will create a simple dataset where exercise affects both weight loss and blood pressure.

Exercise raises weight loss by 0.5 units (on a standardised scale); each unit of weight loss reduces blood pressure by 2 mmHg (millimetres of mercury, the standard clinical unit for blood pressure). There is no direct path from Exercise to Blood pressure: the entire effect is mediated.

\[\text{True total effect of Exercise (Exercise} \rightarrow \text{Blood Pressure)} = 0.5 \times (-2) = -1 \text{ mmHg}\]

\[\text{True direct effect of Exercise (Exercise} \rightarrow \text{Blood Pressure | WeightLoss)} = 0\]

\[\text{True mediated path (WeightLoss} \rightarrow \text{Blood Pressure | Exercise)} = -2 \text{ mmHg per unit}\]

Analyze the data to see the associations

We fit two Bayesian models with Pyro. The only difference is whether WeightLoss appears as a predictor.

Model 1 — total effect (Exercise only):

\[\alpha \sim \mathcal{N}(120,\, 10) \]

\[\beta_{\text{ex}} \sim \mathcal{N}(0,\, 5) \]

\[\sigma \sim \text{HalfNormal}(10)\]

\[\mu_i = \alpha + \underbrace{\beta_{\text{ex}}}_{\text{total effect}} \cdot \text{Exercise}_i \]

\[\text{BP}_i \sim \mathcal{N}(\mu_i,\, \sigma)\]

Model 2 — direct effect (Exercise and WeightLoss):

\[\alpha \sim \mathcal{N}(120,\, 10)\] \[ \beta_{\text{ex}} \sim \mathcal{N}(0,\, 5) \]

\[\beta_{\text{wl}} \sim \mathcal{N}(0,\, 5) \]

\[\sigma \sim \text{HalfNormal}(10)\]

\[\mu_i = \alpha + \underbrace{\beta_{\text{ex}}}_{\text{direct effect}} \cdot \text{Exercise}_i + \underbrace{\beta_{\text{wl}}}_{\text{mediated effect}} \cdot \text{WeightLoss}_i \]

\[\text{BP}_i \sim \mathcal{N}(\mu_i,\, \sigma)\]

In Model 1, \(\beta_{\text{ex}}\) captures the total effect of exercise: that is, everything that reaches blood pressure through any path, including through weight loss.

In Model 2, conditioning on WeightLoss blocks the mediated path, so \(\beta_{\text{ex}}\) estimates only the direct effect that bypasses the mediator, and \(\beta_{\text{wl}}\) isolates the weight-loss-to-blood-pressure relationship.

Code 
# Model 1: total effect (no conditioning on WeightLoss) 
def total_effect_model(exercise, bp):
    alpha    = pyro.sample("alpha",   dist.Normal(120.0, 10.0))
    beta_ex  = pyro.sample("beta_ex", dist.Normal(0.0,   5.0))
    sigma    = pyro.sample("sigma",   dist.HalfNormal(10.0))
    mu = alpha + beta_ex * exercise
    with pyro.plate("obs", len(bp)):
        pyro.sample("bp", dist.Normal(mu, sigma), obs=bp)


# Model 2: direct effect (conditioning on WeightLoss) 
def direct_effect_model(exercise, weight_loss, bp):
    alpha    = pyro.sample("alpha",   dist.Normal(120.0, 10.0))
    beta_ex  = pyro.sample("beta_ex", dist.Normal(0.0,   5.0))
    beta_wl  = pyro.sample("beta_wl", dist.Normal(0.0,   5.0))
    sigma    = pyro.sample("sigma",   dist.HalfNormal(10.0))
    mu = alpha + beta_ex * exercise + beta_wl * weight_loss
    with pyro.plate("obs", len(bp)):
        pyro.sample("bp", dist.Normal(mu, sigma), obs=bp)

                warmup_steps=warmup_steps, disable_progbar=True)

The fitted posteriors make a clear distinction between total effect, direct effect, and mediated path.

For the total effect of Exercise on Blood pressure, the posterior median sits very close to the true value of \(-1\) mmHg. This is the quantity we recover when we do not condition on WeightLoss, because we want to keep the whole causal route

\[\text{Exercise} \rightarrow \text{WeightLoss} \rightarrow \text{Blood pressure}\]

inside the estimand. In other words, this model asks for the overall effect of Exercise on Blood pressure, including the part that works through WeightLoss.

For the direct effect of Exercise on Blood pressure conditional on WeightLoss, the posterior is much more uncertain and is centred somewhat to the right of zero. But the 95% interval still includes \(0\), which is the true direct effect in this simulation. This is exactly the question:

If two people had the same WeightLoss, would Exercise still predict a difference in Blood pressure?

In the data-generating process used here, the answer is no. So once we condition on WeightLoss, the remaining exercise coefficient should be interpreted as a noisy estimate of a direct effect that is truly absent.

For the mediated path coefficient, the posterior is tight and close to the true value of \(-2\) mmHg. This coefficient measures the relationship between WeightLoss and Blood pressure after accounting for Exercise. Because WeightLoss is a continuous predictor with a strong effect, the model learns this parameter much more precisely than either of the Exercise coefficients.

The key point is that the two Exercise coefficients are different because they answer different causal questions. Leaving WeightLoss out gives the total effect of Exercise. Putting WeightLoss into the model blocks the mediation route and gives the direct effect instead.

Conditioning on the mediator does not make the model better or worse. It changes the question. If WeightLoss is part of the mechanism we want to measure, we should leave it out and estimate the total effect. If we want the part of the Exercise effect that does not run through WeightLoss, we should condition on it and estimate the direct effect.

The third is the collider structure:

In a collider, two arrows meet at the same node:

\[X \rightarrow Z \leftarrow Y\]

Here, \(Z\) is a common effect of \(X\) and \(Y\). This is very different from a fork. In a fork, the middle node is a common cause and conditioning on it blocks a confounding path. In a collider, the middle node is a common effect and conditioning on it opens a path that was previously closed.

Without conditioning on \(Z\), the path between \(X\) and \(Y\) is closed:

\[X \perp\!\!\!\perp Y\]

But once we condition on the collider, information can flow through it:

\[X \not\!\perp\!\!\!\perp Y \mid Z\]

This can create an association between \(X\) and \(Y\) even when neither causes the other. The association is not evidence of a causal relationship between the endpoints; it is a consequence of conditioning on a shared outcome. This is called collider bias; when the conditioning happens through sample inclusion, this is often called selection bias.

The practical rule is: do not condition on a collider, or on a variable downstream of a collider, unless the estimand specifically requires it and the induced bias is understood.

TipIntuitive example of a collider structure

Suppose a selective university admits students who are either smart or athletic, or some combination of the two. In the general population, assume intelligence and athleticism are unrelated: knowing that someone is smart tells us nothing about whether they are athletic.

The admission decision is therefore a shared effect:

\[ \text{Smart} \rightarrow \text{Admitted} \leftarrow \text{Athletic}\]

If we look only at admitted students, we have conditioned on the collider. This can create a negative association between intelligence and athleticism even though the two traits were independent in the wider population. Among admitted students, someone who is not athletic is more likely to be very smart, because they still had to clear the admissions threshold somehow.

This is the collider intuition: once we know the shared outcome occurred, learning that one possible cause is absent makes the other cause more plausible. The two causes begin to “explain away” each other.

The observed trade-off is not evidence that intelligence reduces athleticism, or that athleticism reduces intelligence. It is a selection artifact created by conditioning on their shared outcome.

Analyze the data to see the associations

So, should we condition on the collider? Usually, no. If the goal is to learn the relationship between the two causes, Smart and Athletic, in the population, conditioning on Admitted would create an association that is not causal.

In the regression examples in this post, conditioning on a variable means including it as a covariate in the model. So here, conditioning on Admitted means adding Admitted to the linear predictor.

We fit two Bayesian linear regressions: one unconditioned model and one regression-adjusted model that includes Admitted.

Model 1 - no conditioning on the collider:

\[\alpha \sim \mathcal{N}(0,\, 1)\]

\[\beta_{\text{smart}} \sim \mathcal{N}(0,\, 1)\]

\[\sigma \sim \text{HalfNormal}(1)\]

\[\mu_i = \alpha + \beta_{\text{smart}} \cdot \text{Smart}_i\]

\[\text{Athletic}_i \sim \mathcal{N}(\mu_i,\, \sigma)\]

This model uses the full population. Since Smart and Athletic were generated independently, the posterior for \(\beta_{\text{smart}}\) should be close to zero.

Model 2 - conditioning on the collider:

\[\alpha \sim \mathcal{N}(0,\, 1)\]

\[\beta_{\text{smart}} \sim \mathcal{N}(0,\, 1)\]

\[\beta_{\text{admitted}} \sim \mathcal{N}(0,\, 1)\]

\[\sigma \sim \text{HalfNormal}(1)\]

\[\mu_i = \alpha + \beta_{\text{smart}} \cdot \text{Smart}_i + \beta_{\text{admitted}} \cdot \text{Admitted}_i\]

\[\text{Athletic}_i \sim \mathcal{N}(\mu_i,\, \sigma)\]

This model uses the full population but includes the collider as a covariate. The coefficient \(\beta_{\text{smart}}\) is now a conditional association: how Athletic changes with Smart among observations with the same value of Admitted, averaged across the admitted and not-admitted strata by the model.

If the collider logic is correct, the posterior for \(\beta_{\text{smart}}\) should move away from zero in the conditioned model, even though Smart has no causal effect on Athletic.

def smart_athletic_model(smart, athletic, admitted=None):
    alpha = pyro.sample("alpha", dist.Normal(0.0, 1.0))
    beta_smart = pyro.sample("beta_smart", dist.Normal(0.0, 1.0))
    sigma = pyro.sample("sigma", dist.HalfNormal(1.0))

    mu = alpha + beta_smart * smart
    if admitted is not None:
        beta_admitted = pyro.sample("beta_admitted", dist.Normal(0.0, 1.0))
        mu = mu + beta_admitted * admitted

    with pyro.plate("obs", len(athletic)):
        pyro.sample("athletic", dist.Normal(mu, sigma), obs=athletic)

The two posteriors answer different questions.

In the full population, the posterior for \(\beta_{\text{smart}}\) is close to zero. This is what we expect, because Smart and Athletic were generated independently. The path

\[\text{Smart} \rightarrow \text{Admitted} \leftarrow \text{Athletic}\]

is closed until we condition on Admitted.

When we include Admitted as a covariate, the posterior for \(\beta_{\text{smart}}\) moves away from zero. This is regression conditioning: the model asks whether Smart and Athletic are associated after holding admission status fixed. Since admission is a common effect of both traits, holding it fixed opens the collider path. The resulting association is not a causal effect of Smart on Athletic, or of Athletic on Smart. It is created by conditioning on their shared outcome.

So the practical answer is:

  • If the estimand is the relationship between Smart and Athletic in the full population, do not condition on Admitted.
  • If the estimand is a descriptive association at fixed admission status, then including Admitted as a covariate is conditioning on the collider.

For a collider, conditioning is not a harmless adjustment. It changes a closed path into an open path and can manufacture an association between two variables that do not cause each other.

The fourth and final structure is the descendant structure:

A descendant is any variable that lies downstream of another variable. In the structure here, \(D\) is a descendant of the collider \(Z\):

\[X \rightarrow Z \leftarrow Y\]

\[Z \rightarrow D\]

Because \(D\) is caused by \(Z\), learning about \(D\) gives us information about \(Z\). So even if we do not condition directly on the collider \(Z\), conditioning on its descendant \(D\) can still open the otherwise closed path between \(X\) and \(Y\):

\[X \not\!\perp\!\!\!\perp Y \mid D\]

Conditioning on a descendant of a collider can open the collider path indirectly. The problem is not only conditioning on the shared outcome itself, but also conditioning on variables that reveal information about that shared outcome.

TipIntuitive example of a descendant structure

Return to the university admissions example. Suppose Smart and Athletic are independent traits in the applicant population, but both can help a student get admitted:

\[\text{Smart} \rightarrow \text{Admitted} \leftarrow \text{Athletic}\]

Now add one more variable: whether the student appears in the university’s student database. A student appears in that database only after being admitted and enrolling. So database membership is downstream of admission:

\[\text{Admitted} \rightarrow \text{In university database}\]

Admitted is still the collider. In university database is not itself the collider; it is a downstream consequence of it. If we analyze only people in the database, we are using a variable that carries information about admission, so Smart and Athletic can still begin to explain away each other:

\[\text{Smart} \not\!\perp\!\!\!\perp \text{Athletic} \mid \text{In university database}\]

Conditioning on a downstream variable can act like a noisy version of conditioning on the collider itself. The more informative the descendant is about the collider, the more it can induce the same selection-driven association.

Let’s simulate some data to illustrate the descendant example. We will create a simple dataset where Smart and Athletic are independent traits in the applicant population, but both can help a student get admitted. We will then add one more variable: whether the student appears in the university’s student database. Database membership is downstream of admission, but it is noisy: some admitted students are missing because it takes time to add them to the system, and a small number of non-admitted applicants can appear because of administrative errors.

Code 
# Descendant example: database membership as a noisy proxy for admission
rng_descendant = np.random.default_rng(123)

n_applicants = 5000

# Controls how noisy the descendant is.
# Higher values mean more admitted students are still missing from the database.
database_delay_noise = 0.30

# Optional administrative/data-entry noise.
database_false_positive_rate = 0.01

smart = rng_descendant.normal(0, 1, n_applicants)
athletic = rng_descendant.normal(0, 1, n_applicants)

# Admission is the collider: both traits can help a student clear the threshold.
admission_score = smart + athletic
admission_threshold = np.quantile(admission_score, 0.80)
admitted = (admission_score >= admission_threshold).astype(int)

# Database membership is a descendant of admission.
# Because adding admitted students takes time, some admitted students are not
# yet visible in the database. A small false-positive rate represents occasional
# administrative errors.
p_in_database = np.where(
    admitted == 1,
    1.0 - database_delay_noise,
    database_false_positive_rate,
)
in_university_database = rng_descendant.binomial(1, p_in_database)

descendant_df = pd.DataFrame({
    "Smart": smart,
    "Athletic": athletic,
    "Admitted": admitted,
    "InUniversityDatabase": in_university_database,
})

How do path structures guide conditioning decisions for \(s\)?

Here, one important role of the DAG is to help us decide which variables it makes sense to condition on, and which ones we should avoid conditioning on, when estimating the target-binding propensity \(s\). To do this, we follow the recipe suggested by McElreath in Chapter 6 of the second edition of Statistical Rethinking:

In the statistical model, conditioning on a variable usually means including it as an observed predictor or covariate. In the DAG, the same choice has a causal interpretation: it can block some paths and open others. So the question is not only whether a variable improves prediction, but whether conditioning on it changes the causal path we are trying to learn from.

  1. List all paths connecting \(s\) and \(C_t\).

  2. Classify each path as open or closed. Conditioning on a non-collider blocks a path, while conditioning on a collider, or on one of its descendants, opens a path that would otherwise be closed.

  3. Identify whether any open path is a backdoor path. A backdoor path has an arrow entering \(s\) and can carry confounding rather than the target-binding signal we want.

  4. Decide which variables should be conditioned on. Common causes usually need to be adjusted for, mediators depend on whether we want total or direct effects, and colliders should usually be left unconditioned.

For the DEL model, this means we should not ask only whether a variable is predictive of \(C_t\). We should ask what role it plays in the DAG. A variable can be predictive because it is a cause, a mediator, a confounder, or a collider. These cases imply different modelling choices.

The DAG turns variable selection into a causal question: condition on variables that block unwanted backdoor paths, avoid conditioning on colliders, and condition on mediators only when the target estimand is a direct effect.
